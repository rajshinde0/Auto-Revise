SECTION 1: MULTI-LEVEL INDEXING
================================================================================

DEFINITION:
Multi-level indexing is a hierarchical indexing technique where indexes are 
created on top of other indexes, forming multiple levels of indirection. This 
dramatically reduces the number of disk accesses required to locate a record.

--------------------------------------------------------------------------------
THE PROBLEM IT SOLVES:
--------------------------------------------------------------------------------

Scenario: Table with 1,000,000 records
- Without Index: Linear search requires up to 1,000,000 disk I/O operations
- With Single-Level Index: Still requires ~1,000 disk I/O operations
- Issue: When the index itself becomes too large to fit in memory

Solution: Create an index on the index (and repeat if necessary)

--------------------------------------------------------------------------------
HIERARCHICAL STRUCTURE:
--------------------------------------------------------------------------------

Level 3 (Top):        [Master Index]              ← Smallest, fits in RAM
                      /      |      \
                     /       |       \
Level 2:        [Index]  [Index]  [Index]         ← Secondary indexes
                /    \    /    \    /    \
Level 1:     [Idx] [Idx] [Idx] [Idx] [Idx]       ← Primary indexes
             /  \   /  \   /  \   /  \
Level 0:  [Data][Data][Data][Data][Data]         ← Actual records on disk

--------------------------------------------------------------------------------
HOW IT WORKS - STEP BY STEP:
--------------------------------------------------------------------------------

1. Top-Level Index (Level 3):
   - Contains few entries (100-1000)
   - Stored in main memory (RAM)
   - Points to Level 2 index blocks

2. Mid-Level Indexes (Level 2):
   - Contains more entries (10,000-100,000)
   - Stored on disk
   - Points to Level 1 index blocks

3. Base-Level Index (Level 1):
   - Contains all key values
   - Points to actual data blocks

4. Data Level (Level 0):
   - Actual table records stored on disk

--------------------------------------------------------------------------------
SEARCH ALGORITHM:
--------------------------------------------------------------------------------

To find record with key = K:

Step 1: Load top-level index into memory (1 disk I/O)
Step 2: Binary search in memory to find pointer to Level 2 block
Step 3: Load appropriate Level 2 block (1 disk I/O)
Step 4: Binary search to find pointer to Level 1 block
Step 5: Load appropriate Level 1 block (1 disk I/O)
Step 6: Binary search to find pointer to data block
Step 7: Load data block (1 disk I/O)

Total Disk I/O: 4 operations (instead of 1,000,000!)

--------------------------------------------------------------------------------
MATHEMATICAL ANALYSIS:
--------------------------------------------------------------------------------

Given:
- n = total number of records
- b = blocking factor (entries per index block)
- Levels required = ⌈log_b(n)⌉

Example with 1,000,000 records and blocking factor 100:
- Level 0: 1,000,000 data records
- Level 1: 10,000 index entries (1,000,000 / 100)
- Level 2: 100 index entries (10,000 / 100)
- Level 3: 1 index entry (100 / 100)

Disk I/O = 4 (one per level)

Without multi-level indexing:
- Best case: 1 I/O (lucky first try)
- Average case: 500,000 I/O
- Worst case: 1,000,000 I/O

With multi-level indexing:
- Best/Average/Worst case: 4 I/O (consistent!)

Time Complexity: O(log_b n) where b is blocking factor

--------------------------------------------------------------------------------
ADVANTAGES:
--------------------------------------------------------------------------------

1. REDUCED DISK I/O:
   - Logarithmic search time instead of linear
   - Top levels cached in memory

2. EFFICIENT FOR LARGE DATABASES:
   - Scales well with millions/billions of records
   - Height grows logarithmically, not linearly

3. DYNAMIC:
   - Can add/remove levels as data grows/shrinks
   - Self-organizing structure

4. MEMORY EFFICIENT:
   - Only top level needs to be in RAM
   - Lower levels loaded on-demand

--------------------------------------------------------------------------------
DISADVANTAGES:
--------------------------------------------------------------------------------

1. INSERTION/DELETION OVERHEAD:
   - Must update multiple index levels
   - Requires cascading updates

2. STORAGE OVERHEAD:
   - Indexes themselves consume disk space
   - Multiple levels multiply storage cost

3. MAINTENANCE COMPLEXITY:
   - Index rebuilding needed after bulk operations
   - Fragmentation over time

SECTION 2: B+ TREE (BALANCED PLUS TREE)
================================================================================

DEFINITION:
B+ Tree is a self-balancing, multi-level tree data structure that maintains 
sorted data and allows searches, sequential access, insertions, and deletions 
in logarithmic time. It is the PRIMARY indexing structure used by MySQL InnoDB.

--------------------------------------------------------------------------------
STRUCTURAL PROPERTIES:
--------------------------------------------------------------------------------

1. MULTI-WAY TREE:
   - Each node can have multiple children (not just 2 like binary tree)
   - Order m: Maximum m children per node
   - High fanout reduces tree height

2. BALANCED:
   - All leaf nodes at the same depth
   - Guarantees consistent O(log n) performance
   - Self-balancing through splits and merges

3. SORTED:
   - Keys within nodes are sorted
   - Keys in subtree[i] < key[i] < keys in subtree[i+1]

4. DATA ONLY IN LEAVES:
   - Internal nodes: Keys + pointers (for navigation)
   - Leaf nodes: Keys + data (or pointers to data)
   - This is the "+" in B+ Tree

5. LINKED LEAVES:
   - Leaf nodes form a doubly-linked list
   - Enables efficient range queries and sequential scans

--------------------------------------------------------------------------------
NODE STRUCTURE:
--------------------------------------------------------------------------------

INTERNAL NODE (Non-Leaf):
┌─────────────────────────────────────────┐
│ [P₀|K₁|P₁|K₂|P₂|...|Kₙ|Pₙ]            │
└─────────────────────────────────────────┘
Where:
- Pᵢ = Pointer to child node
- Kᵢ = Key value (separator)
- Keys: K₁ < K₂ < ... < Kₙ
- n ≤ m (order of tree)

LEAF NODE:
┌─────────────────────────────────────────┐
│ [K₁|D₁|K₂|D₂|...|Kₙ|Dₙ] → [Next Leaf] │
└─────────────────────────────────────────┘
Where:
- Kᵢ = Key value
- Dᵢ = Data record or pointer to data
- → = Pointer to next leaf (for sequential access)

--------------------------------------------------------------------------------
COMPLETE B+ TREE EXAMPLE:
--------------------------------------------------------------------------------

Order m = 4 (max 4 children per node)
Keys: 5, 10, 15, 20, 25, 30, 35, 40, 45, 50

                         [25]                    ← ROOT (Level 2)
                    /            \
             [10, 20]          [35, 45]          ← INTERNAL (Level 1)
            /    |    \        /    |    \
         [5]  [10,15] [20] [25,30] [35,40] [45,50]  ← LEAVES (Level 0)
          ↕      ↕      ↕     ↕      ↕       ↕
        Data   Data   Data  Data   Data    Data

Linked list at leaf level:
[5] ←→ [10,15] ←→ [20] ←→ [25,30] ←→ [35,40] ←→ [45,50]

--------------------------------------------------------------------------------
SEARCH ALGORITHM IN B+ TREE:
--------------------------------------------------------------------------------

To search for key K:

1. Start at root node
2. While current node is not a leaf:
   a. Find smallest key Kᵢ where K ≤ Kᵢ
   b. Follow pointer Pᵢ to child node
3. At leaf node:
   a. Search for key K
   b. If found: return data Dᵢ
   c. If not found: return NULL

EXAMPLE: Search for key = 37 in above tree

Step 1: At root [25]
        37 > 25, go right

Step 2: At internal [35, 45]
        35 ≤ 37 < 45, go middle

Step 3: At leaf [35, 40]
        37 not found exactly, but range is correct
        Return: "Not found" or "Between 35 and 40"

Disk I/O: 3 operations (tree height = 3)

--------------------------------------------------------------------------------
RANGE QUERY (B+ TREE ADVANTAGE):
--------------------------------------------------------------------------------

Query: SELECT * FROM Users WHERE user_id BETWEEN 15 AND 40

Step 1: Search for key = 15 (find starting leaf)
        Navigate: Root → [10,20] → Leaf [10,15]

Step 2: Follow linked list from [10,15]:
        [10,15] → [20] → [25,30] → [35,40]
        Collect all keys in range: 15, 20, 25, 30, 35, 40

Step 3: Stop when key > 40

Efficiency: Only traverse relevant leaves (no backtracking!)

--------------------------------------------------------------------------------
INSERTION ALGORITHM:
--------------------------------------------------------------------------------

To insert key K with data D:

1. Search for leaf node L where K belongs
2. If L has space (not full):
   - Insert K and D in sorted order
3. If L is full (overflow):
   - Split L into L and L'
   - Distribute keys evenly
   - Copy middle key up to parent
   - If parent full, recursively split upward
4. Tree grows upward from root (stays balanced!)

EXAMPLE: Insert 17 into previous tree

Before:        [10,15] [20]
After split:   [10,15,17] → Full! Split needed
Result:        [10,15] [17,20]
Parent update: Add separator 17 to internal node

--------------------------------------------------------------------------------
DELETION ALGORITHM:
--------------------------------------------------------------------------------

To delete key K:

1. Search for leaf node L containing K
2. Remove K from L
3. If L becomes underfull (< ⌈m/2⌉ keys):
   a. Try to borrow from sibling
   b. If sibling also minimal, merge with sibling
   c. Update parent (recursive if needed)
4. Tree shrinks from root if needed

Maintains balance through merging and redistribution.

--------------------------------------------------------------------------------
B+ TREE PROPERTIES (FORMAL):
--------------------------------------------------------------------------------

For a B+ Tree of order m:

1. Root:
   - Has at least 2 children (unless it's a leaf)
   - Has at most m children

2. Internal Nodes:
   - Have at least ⌈m/2⌉ children
   - Have at most m children
   - With k children, have k-1 keys

3. Leaf Nodes:
   - Have at least ⌈m/2⌉ keys
   - Have at most m-1 keys
   - All at same depth

4. Height:
   - h ≤ ⌈log₍ₘ/₂₎(n)⌉ where n = number of keys
   - Typically 3-4 levels for millions of records

--------------------------------------------------------------------------------
WHY MYSQL INNODB USES B+ TREES:
--------------------------------------------------------------------------------

1. DISK-ORIENTED DESIGN:
   - Nodes sized to match disk block size (16KB in InnoDB)
   - High fanout minimizes tree height
   - Each disk I/O fetches entire node (many keys at once)

2. EXCELLENT FOR RANGE QUERIES:
   - Linked leaves enable sequential scanning
   - No need to traverse tree multiple times
   - ORDER BY and BETWEEN clauses are fast

3. CONSISTENT PERFORMANCE:
   - All operations O(log n)
   - No worst-case degradation (unlike hash indexes)
   - Self-balancing maintains guarantee

4. EFFICIENT CACHING:
   - Upper levels stay in buffer pool (RAM)
   - Only leaves require disk access
   - Reduces average I/O to 1-2 operations

5. SUPPORTS BOTH SEEKS AND SCANS:
   - Point queries: Navigate tree
   - Range queries: Scan linked leaves
   - Full table scan: Traverse leaf linked list

--------------------------------------------------------------------------------
QUERY EXECUTION WITH B+ TREES:
--------------------------------------------------------------------------------

EXAMPLE 1: Point Query
```sql
SELECT * FROM Users WHERE user_id = 12345;
```

Execution Plan:
1. Access Users primary key B+ tree
2. Navigate: Root (Level 2) → Internal (Level 1) → Leaf (Level 0)
3. Disk I/O: 3 operations
4. Time: O(log n) = O(log 1000000) ≈ 20 comparisons

EXAMPLE 2: Range Query with Secondary Index
```sql
SELECT * FROM CardPerformance 
WHERE user_id = 100 
  AND next_review_date <= '2025-12-03';
```

Execution Plan:
1. Access composite index (user_id, card_id)
2. Navigate to user_id = 100 (3 disk I/O)
3. Scan linked leaves for all cards of user 100 (sequential)
4. For each match, check next_review_date (in-memory filter)
5. Lookup primary key for full record (clustered index)
6. Total I/O: 3 + k (where k = matching records)

EXAMPLE 3: JOIN with Multiple B+ Trees
```sql
SELECT c.*, d.deck_name 
FROM Cards c 
JOIN Decks d ON c.deck_id = d.deck_id 
WHERE c.card_id BETWEEN 1000 AND 2000;
```

Execution Plan:
1. Access Cards primary key B+ tree
2. Navigate to card_id = 1000
3. Scan linked leaves: 1000 → 1001 → ... → 2000
4. For each card, lookup deck_id in Decks B+ tree
5. Nested loop join using two B+ trees
6. Total I/O: 3 + 1000 + (1000 * 3) ≈ 4000 operations

Optimization: Add index on Cards(deck_id) to avoid full Decks scan

--------------------------------------------------------------------------------
MULTI-LEVEL NATURE IN INNODB B+ TREES:
--------------------------------------------------------------------------------

Typical InnoDB B+ Tree for 1 Million Users:

LEVEL 3 (ROOT):
- 1 node (16KB)
- ~200 keys
- Always in buffer pool (memory)

LEVEL 2 (INTERNAL):
- ~200 nodes (3.2MB)
- ~40,000 keys
- Often in buffer pool

LEVEL 1 (INTERNAL):
- ~40,000 nodes (640MB)
- ~8,000,000 keys
- Partially in buffer pool

LEVEL 0 (LEAVES):
- ~1,000,000 nodes (16GB)
- Actual user records
- Mostly on disk

Average Query:
- Check root in memory (0 I/O)
- Check internal in memory (0 I/O)
- Load leaf from disk (1 I/O)
- Total: 1 disk access!

--------------------------------------------------------------------------------
BUFFER POOL CACHING STRATEGY:
--------------------------------------------------------------------------------

InnoDB Buffer Pool (Default 128MB - 8GB):

Priority 1: Root and top internal nodes
Priority 2: Hot leaf nodes (frequently accessed)
Priority 3: Recently used nodes

LRU (Least Recently Used) eviction policy

Result:
- 95%+ of queries hit cache (0 disk I/O)
- Only cold data requires disk access
- Multi-level B+ tree + caching = extremely fast

SECTION 5: PRACTICAL EXAMPLES IN AUTOREVISE
================================================================================

--------------------------------------------------------------------------------
EXAMPLE 1: Study Session Query
--------------------------------------------------------------------------------

```sql
SELECT c.card_id, c.front_content, c.back_content
FROM Cards c
JOIN Decks d ON c.deck_id = d.deck_id
LEFT JOIN CardPerformance cp ON c.card_id = cp.card_id 
  AND cp.user_id = 123
WHERE d.user_id = 123
  AND (cp.next_review_date IS NULL OR cp.next_review_date <= CURDATE())
LIMIT 20;
```

B+ Tree Usage:
1. Decks(user_id): Secondary B+ tree → Find user's decks (3 I/O)
2. Cards(deck_id): Foreign key B+ tree → Find cards per deck (3 I/O)
3. CardPerformance(user_id, card_id): Composite B+ tree → Check review dates
4. Date filter uses B+ tree range scan on next_review_date

Total: ~10-15 disk I/O (with caching: 1-3 I/O)

Without B+ Trees: 1,000,000+ full table scans!

--------------------------------------------------------------------------------
EXAMPLE 2: Achievement Check Query
--------------------------------------------------------------------------------

```sql
SELECT COUNT(*) as count 
FROM Cards c 
JOIN Decks d ON c.deck_id = d.deck_id 
WHERE d.user_id = 123;
```

B+ Tree Optimization:
1. Decks(user_id): Find all user's decks via B+ tree
2. Cards(deck_id): For each deck, count via B+ tree
3. No need to load full card records (counting index entries only)

Covered Index Optimization: COUNT(*) served entirely from index!

--------------------------------------------------------------------------------
EXAMPLE 3: MCQ Category Browse
--------------------------------------------------------------------------------

```sql
SELECT m.mcq_id, m.question_text, m.difficulty
FROM MCQ_Questions m
WHERE m.category_id = 5
ORDER BY m.created_at DESC
LIMIT 50;
```

B+ Tree Usage:
1. MCQ_Questions(category_id): B+ tree navigates to category=5
2. Linked leaf traversal collects all questions in category
3. Sort by created_at: Additional index scan or filesort

Optimization: Composite index on (category_id, created_at) eliminates sort!

SECTION 6: OPTIMIZATION RECOMMENDATIONS
================================================================================

Based on B+ Tree characteristics:

1. CREATE COMPOSITE INDEXES for multi-column WHERE clauses:
   ```sql
   CREATE INDEX idx_performance_review 
   ON CardPerformance(user_id, next_review_date);
   ```

2. USE COVERING INDEXES to avoid lookups:
   ```sql
   CREATE INDEX idx_cards_deck_content 
   ON Cards(deck_id, front_content, back_content);
   ```

3. INDEX FOREIGN KEYS (InnoDB does this automatically):
   - Speeds up JOIN operations
   - Enables merge join algorithm

4. AVOID FUNCTIONS ON INDEXED COLUMNS (breaks B+ tree):
   ❌ WHERE YEAR(created_at) = 2025
   ✓ WHERE created_at BETWEEN '2025-01-01' AND '2025-12-31'

5. ORDER MATTERS IN COMPOSITE INDEXES:
   - Index (user_id, date) helps: WHERE user_id=X AND date=Y
   - Doesn't help: WHERE date=Y (leftmost prefix rule)

6. MONITOR INDEX USAGE:
   ```sql
   EXPLAIN SELECT * FROM CardPerformance WHERE user_id = 123;
   ```

7. REBUILD FRAGMENTED INDEXES:
   ```sql
   OPTIMIZE TABLE CardPerformance;
   ```
